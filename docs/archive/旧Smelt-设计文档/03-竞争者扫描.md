# 竞争者扫描（2026-07-17）

## 残酷的事实：不是第一个

截至 2026 年 7 月，已有至少 8 个项目在做"把人+AI 协作打包成可移植文件"这件事。

---

## 最接近的竞争者

### CLAN（最危险）
- GitHub: `saieeshward/clan`
- 一句话：**"不是框架，是文件格式——agent 到 agent 的交接"**
- ZIP 包，内含 manifest.yaml、决策链、输出 schema、人类可读 HTML
- 核心卖点：provenance 强制（CLI 拒绝无 `--agent` 和 `--action` 的写入）、并行 agent 写入独立命名空间零冲突、零 LLM 确定性合并
- 没有运行时、没有云端、没有平台锁定。任何语言都能实现。开源 MPL-2.0
- 已有 Rust CLI、186 个测试、跨平台二进制
- **和 .card 的差异**：面向多 agent 流水线，不是面向普通用户的工作台。用 CLI 操作而非拖拽卡片。没有"空间"概念。没有隐私编辑。

### Capsules.Run
- 一句话：**"可移植的智能单元"**
- `.capsule` 文件 = 内容 + 状态 + 可验证事件账本
- 支持模板（命名的格式对应真实工作流）、生命周期阶段、MCP 集成
- 可直接丢进任何 LLM，带一个 bootstrap prompt 就能用
- 目前 program.md 是核心产出（markdown），未来扩展到任意媒体类型
- **和 .card 的差异**：更偏向 LLM-to-LLM 协作，没有视觉工作台、没有拖拽、没有隐私编辑流程

### AGX（Agent Graph eXchange）
- GitHub: `agx-dev/agx`
- Docker 类比：ACP 协议 = OCI 规范，`.agx` = 容器镜像，`agxd` = containerd
- tar+zstd 压缩，包含 manifest、identity、memory（情景/语义/程序）、工具定义、策略、签名
- 模型无关，版本化，可审计
- 和 MCP/A2A 协议栈集成
- **和 .card 的差异**：太重了。目标是 agent 的 Docker 化，不是普通用户的日常协作封装

### .agent（agentpk.io）
- 一句话：**"AI agent 的 .whl"**
- ZIP 归档，包含 manifest + 代码 + trust score（0-100，证明 manifest 是否诚实）
- 框架无关，支持签名、版本对比
- **和 .card 的差异**：打包的是 agent 本身（代码+配置），不是"一次协作过程"

### Agent Session Protocol
- GitHub: `kevin-dp/agent-session-protocol`
- 可移植 JSONL 格式，用于跨 agent 共享和恢复编码会话
- 支持 Claude Code 和 Codex，带 CLI 工具（`capi`）
- 可以实时协作（浏览器提交 prompt 通过 MCP 通道）
- **和 .card 的差异**：只管对话 session 的跨 agent 迁移，不管产物、不管隐私编辑、不管视觉化

### ctxpkg
- 一句话：**"给 AI agent context 代码级别的严谨——打包、版本化、验证、可移植"**
- 单个 JSON 文件，五层类型内容（knowledge、gotchas、graph、session、policies）
- SHA-256 完整性校验，SemVer 版本化
- **和 .card 的差异**：面向 agent 的知识包，不是人+AI 协作过程的完整记录

### AIFP（AI First Project Format）
- ZIP 容器，打包完整 AI 生成项目：音频、视频、图像、prompt、脚本、时间线、元数据、来源
- 面向多媒体创作（Suno + Veo + Stable Diffusion 组合）
- **和 .card 的差异**：面向 AI 生成的多媒体项目，不是通用的"一次协作"

### OACP（Open Agent Coordination Protocol）
- 基于文件的 agent 间通信协议。12 种类型化消息。共享持久记忆。
- 文件系统即总线，无后台进程，无中心服务器
- **和 .card 的差异**：是通信协议，不是文件格式。解决的是"agent 之间怎么说话"，不是"协作过程怎么打包"

## 关键观察

### .card 没有被任何人完全覆盖的组合

| 特征 | CLAN | Capsules | AGX | .agent | Session Protocol | .card |
|---|---|---|---|---|---|---|
| 可移植文件格式 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 无运行时/平台锁定 | ✅ | ✅ | ❌ | ✅ | ✅ | ✅ |
| 过程可追溯 | ✅ | ✅ | ✅ | ❌ | ✅ | ✅ |
| 面向普通用户（非开发者） | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |
| 视觉工作台（卡片+拖拽） | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |
| 隐私编辑（分享前删减） | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |
| 格式即程序（主题/插件） | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |
| 联邦式（任何空间可打开） | 〰️ | 〰️ | ❌ | ✅ | ✅ | ✅ |
| 签名+加密可选层 | ❌ | ✅ | ✅ | ✅ | ❌ | ✅ |

### 几乎所有人都忽视了的东西

**1. 用户界面。** 所有竞争者都是 CLI 工具或 SDK。没有人做拖拽式工作台。没有人把"卡片"作为交互模型。它们全部面向开发者。

**2. 隐私编辑。** 没有人在导出阶段提供"删掉这段对话""隐藏这个 API key"的能力。所有格式都是 dump 全部内容。

**3. 格式即程序。** 没有人把同一个 .card 格式同时用于"协作产出"和"空间扩展（主题/插件）"。CLAN 最接近——它把 spec 嵌入每个文件——但它不做 UI 扩展。

**4. 普通用户。"** 把数据表拖进去、AI 帮我分析、导出 word、电脑干干净净"——这个体验，目前没有任何竞争者在做。所有人都在为 agent 和开发者做格式。没有人问"一个做数据分析的普通大学生怎么用"。

## 结论

不是第一个想到"把人+AI 协作打包成文件"的人。

但是第一个想到**"把人+AI 协作打包成文件，然后让一个不会写代码的人也能用、也能分享、也能继承、也能拿它装修自己的空间"**的人。

在格式层，竞争者已经存在。在**体验层和用户层**，还没有人入场。

这决定了战略：格式规范要兼容和超越已有标准（不能另起炉灶），但产品形态（视觉工作台 + 卡片交互 + 隐私编辑 + 联邦空间）是空白地带。真正的壁垒不在 .card 的编码方式——竞争者也用了 ZIP+JSON——而在**用户用 .card 的方式**。

要做的事不是发明一个新编码格式。是**做一个让普通人第一次用就理解"这是什么"的空间。**
