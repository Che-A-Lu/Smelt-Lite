# .card 格式规范 v2

> 阿澈 + Dalu，2026-07-19。
> v1（FORMAT.md）定义了容器结构。v2 定义完整链条：AI 对话 → 结构化 JSON → .card → 数字资产。

---

## 零、核心原则

**.card 不是改名 zip。** 一个 .card 跟 .zip 的区别在于：收到的人不用任何人解释，打开就知道这是什么、谁做的、经过谁的手、做到哪一步、能不能接着做。

实现这条原则的方式：**两个 JSON 格式 + 三段标准提示词。**

两个 JSON 的分工：

| | summary.json | trail.jsonl |
|---|---|---|
| 给谁用 | **人**（快速理解全貌） | **下一个 AI**（从断点继续工作） |
| 叫什么 | 战报 | 痕迹 |
| 内容 | 摘要 + 进度 + 转折 + 决策 | 状态变更点 + 触发变更的关键原文 |
| 大小 | ~2KB | ~5KB |
| 谁生成 | AI（对话中或对话后） | AI（对话中或对话后） |

## 一、summary.json（战报——给人看）

### 1.1 完整模板

```jsonc
{
  "format": "card-v1",
  "title": "论文实证分析：AI 跨国差异的人力资本约束",
  "description": "Dalu 和 DeepSeek V4 Pro 协作完成国际贸易实证论文的实证策略阶段。从固定效应出发，经内生性诊断，转向面板工具变量法。",
  "created": "2026-07-19T10:30:00Z",
  "human": {
    "name": "Dalu",
    "role": "定义者——负责方向判断、质量控制、最终决策"
  },
  "ai": {
    "model": "DeepSeek V4 Pro",
    "role": "分析策划师——负责数据诊断、策略建议、漏洞扫描"
  },
  "status": "实证策略选型中",
  "done": "数据收集、预处理、描述性统计",
  "current": "工具变量有效性检验",
  "blocked": null,
  "next": ["弱工具变量检验", "过度识别检验", "2SLS 回归", "写实证结论"],
  "branches": [
    {
      "id": 1,

### 1.2 字段定义

| 字段 | 层级 | 类型 | 说明 |
|---|---|---|---|
| `format` | 必填 | 固定 `"card-v1"` | 格式版本 |
| `title` | 必填 | 字符串 | 这次协作在做什么 |
| `description` | 必填 | 字符串 | AI 自动提取的一句话摘要 |
| `created` | 必填 | ISO 8601 | 创建时间 |
| `human.name` | 必填 | 字符串 | 人的名字 |
| `human.role` | 必填 | 字符串 | 人在协作里的定位 |
| `ai.model` | 必填 | 字符串 | AI 模型名 |
| `ai.role` | 必填 | 字符串 | AI 在协作里的定位 |
| `status` | 必填 | 自由文本 | 当前阶段（如"实证策略选型中"） |
| `done` | 推荐 | 字符串 | 已完成事项，一句话 |
| `current` | 必填 | 字符串 | 当前卡在哪里 |
| `blocked` | 推荐 | 字符串或 null | 阻塞原因，没有就 null |
| `next` | 必填 | 字符串数组 | 下一步要做的事 |
| `branches` | 推荐 | 数组 | 方向变更记录 |
| `branches[].id` | 必填 | 数字 | 转折点编号，对应 trail.jsonl 的 `collab_id` |
| `branches[].what` | 必填 | 字符串 | 从什么转到了什么 |
| `branches[].why` | 必填 | 字符串 | 触发变更的原因 |
| `branches[].who_decided` | 必填 | 字符串 | 谁决定换方向 |
| `decisions` | 推荐 | 数组 | 关键决策记录 |
| `decisions[].id` | 必填 | 数字 | 决策编号 |
| `decisions[].what` | 必填 | 字符串 | 决策内容 |
| `decisions[].why` | 必填 | 字符串 | 决策理由 |
| `decisions[].by` | 必填 | 字符串 | 谁做的决策 |
| `files` | 推荐 | 数组 | 引用的素材文件列表 |
| `files[].name` | 必填 | 字符串 | 文件名 |
| `files[].role` | 必填 | 字符串 | 该文件在项目里的作用 |

## 二、trail.jsonl（痕迹——给下一个 AI）

### 2.1 设计约束

**trail.jsonl 不是对话记录。** 是给下一个 AI 的续作指南。下一个 AI 不需要读完整对话——它只需要知道**什么状态在什么时候被什么原因改变了。**

每条记录一条状态变更。每条记录的 `context` 字段只放**触发变更的关键原文**：
- 最多 3 条消息（一条 human + 一条 AI，必要时加一条）
- 每条消息不超过 150 字
- 不放原文摘要——是原话，但只放最关键的那一两句

### 2.2 每行结构

```jsonl
{"id":"s1","type":"decision","summary":"确定使用面板工具变量法","context":[{"role":"human","text":"固定效应结果不对，直觉告诉我内生性没解决"},{"role":"ai","text":"Hausman 检验 χ²=18.3, p<0.01，确认存在内生性。建议工具变量法。"}],"before":"固定效应模型方案","after":"面板工具变量法方案","confidence":"high","collab_ids":[1]}
{"id":"s2","type":"block","summary":"等待 Dalu 确认工具变量数据可得性","context":[{"role":"ai","text":"需要你确认：CSMAR 里有没有省级政策冲击数据？"},{"role":"human","text":"我去查一下。"}],"before":"工具变量方案讨论中","after":"暂停，等待数据确认","confidence":"high","collab_ids":[]}
```

### 2.3 字段定义

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | 字符串 | 唯一标识，`"s1"`, `"s2"` ... |
| `type` | 枚举 | `branch` / `decision` / `progress` / `block` / `file` |
| `summary` | 字符串 | 这次变更的一句话描述 |
| `context` | 数组 | 触发变更的关键原文，最多 3 条，每条 ≤150 字 |
| `before` | 字符串 | 变更前的状态 |
| `after` | 字符串 | 变更后的状态 |
| `confidence` | 枚举 | AI-1 的自我评估：`high` / `medium` / `low` |
| `collab_ids` | 数组 | 关联 summary.json 里 branches/decisions 的 id |

### 2.4 典型大小

| 项目 | 值 |
|---|---|
| 每条记录 context | 1-3 句关键原文，~300 字 |
| 一个协作过程的状态变更数 | 5-20 次 |
| 总大小 | **2-8 KB** |

## 三、三段标准提示词

这是 Smelt Lite 运转的前提。用户复制提示词 → 贴进任何 AI 工具 → 拿到 JSON → 拖进 Smelt Lite → 打包。

### 3.1 场景 A-开头（省力模式，推荐）

贴在 AI 对话开始前。Token 开销接近零。

```text
## .card 协作记录指令（开头）

在接下来的协作中，请在后台维护一份协作记录。不需要输出。只需要在以下情况发生时在记忆中标注：

- 方向变更（我们换了方法、换了策略、换了思路）
- 关键决策（有人说了"我们决定用 X"）
- 进度推进（一个阶段完成，进入下一阶段）
- 产出文件（新文件产生了，或旧文件被修改了）
- 被阻塞（需要人确认某件事才能继续）

协作结束时我会让你输出结构化 JSON。不用提前输出。
```

### 3.2 场景 A-结尾（省力模式，续）

贴在整个对话结束时。AI 基于全程的记忆直接输出。

```text
## .card 协作记录指令（结尾）

现在请根据你在本次协作中的记忆，输出两份文件。

### 第一份：summary.json

按照以下 JSON 模板输出（用 ```json 代码块包裹）。字段说明：

- format：固定 "card-v1"
- title：协作项目名称
- description：一句话摘要
- created：ISO 8601 时间
- human.name / human.role：人的名字和角色定位
- ai.model / ai.role：你的模型名和你在协作中的角色定位
- status：当前阶段（自由文本）
- done：已完成事项，一句话
- current：当前卡在哪里
- blocked：阻塞原因，没有就填 null
- next：下一步要做的事（数组）
- branches：方向变更记录（数组），每项 {id, what, why, who_decided}
- decisions：关键决策记录（数组），每项 {id, what, why, by}
- files：引用的素材文件（数组），每项 {name, role}

不确定的地方不要编。标记 confidence=low 并注明不确定的原因。

### 第二份：trail.jsonl

每行一条状态变更的 JSON。字段：
- id："s1", "s2" ...
- type："branch" / "decision" / "progress" / "block" / "file"
- summary：这句话描述什么变了
- context：触发变更的关键原文。只放最关键的 1-3 条消息，每条 ≤150 字。是原话，不是摘要。
- before / after：变更前后的状态
- confidence："high" / "medium" / "low"
- collab_ids：关联 summary.json 里 branches/decisions 的 id 数组

trail.jsonl 不是对话记录。只记录状态变更点。维持性对话（追问、解释、确认细节）不记录。
```

### 3.3 场景 B-补救（已聊完、没装开头提示词）

贴在已有对话的结尾。AI 需要重新读全文提取。

```text
## .card 协作记录指令（补救模式）

你需要从我们的对话历史中提取结构化协作记录。请按以下步骤操作：

### 步骤 1：扫描状态变更
通读全文。标出所有「状态变更点」：
- 方向变更（换了方法、策略、思路）
- 关键决策（明确说了"用 X"）
- 进度推进（完成一个阶段）
- 被阻塞（等待确认）
- 产出文件

维持性对话（追问、解释、确认细节）不标。

### 步骤 2：提取关键原文
对每个变更点，找到触发它的那句话。只放 1-3 条消息。每条 ≤150 字。是原话，不是摘要。

### 步骤 3：输出 summary.json + trail.jsonl

按以下规范输出（用 ```json 和 ```jsonl 代码块分别包裹）：

[此处嵌入上面场景 A-结尾的 JSON Schema，内容相同，不再重复]

### 重要约束

- 不确定的标记 confidence=low，不要编造
- trail.jsonl 每条 context 最多 3 条消息，每条 ≤150 字
- 如果全文太长无法一次处理，请分段提取后合并
- 只提取状态变更点。维持性对话不进入 trail.jsonl
```

## 四、.card 的完整生命周期

### 阶段 1：生成 JSON

用户把提示词贴进 AI 工具。AI 输出 summary.json + trail.jsonl。用户保存。

### 阶段 2：打包为 .card

```
打开 smelt-lite 网页
    ↓
拖入 / 粘贴 summary.json（必填）和 trail.jsonl（如果有）
    ↓
（可选）拖入素材文件
    ↓
页面解析 summary.json，展示预览：
  ├── 标题 + 状态 + 描述
  ├── 角色：谁 + 什么 AI
  ├── 进度：完成了什么、卡在哪、下一步
  ├── 转折点：几次方向变更
  └── 文件：待打包的文件清单
    ↓
点「打包为 .card」
    ↓
打包确认面板：
  ├── 签名身份：Dalu (a1b2c3d4)  [切换/导入密钥]
  ├── 时间：自动
  ├── 摘要：自动从 summary.json 提取
  ├── [x] 签名（ECDSA P-256）
  └── [ ] 加密（AES-256-GCM + 密码）
    ↓
下载 .card
```

### 阶段 3：接收方打开 .card

```
拖 .card 进 smelt-lite 网页
    ↓

  │  │  → 李明 修改 · 7/20   │    │
  │  │ 签名：✅ ✅ 全部有效   │    │
  │  └───────────────────────┘    │
  │                              │
  │  ┌─ 做了什么 ───────────┐    │
  │  │ ✅ 数据收集与预处理    │    │
  │  │ 🔄 工具变量有效性检验  │    │

## 五、.card 容器格式

### 5.1 内部目录

```
example.card  (.zip)
├── manifest.json       ← 元信息 + 摘要 + 来源链（Smelt Lite 自动生成）
├── summary.json        ← 战报（用户提供）
├── trail.jsonl         ← 痕迹（用户提供，可选）
├── artifacts/          ← 原始素材
│   ├── main_data.dta
│   └── report.md
├── edits.json          ← 差异记录（每次重新打包追加一条）
├── signature.json      ← ECDSA 签名链
└── README.md           ← 自动生成（改名 .zip 解压的人也能看懂）
```

### 5.2 manifest.json 完整模板

```jsonc
{
  "format": "card-v1",
  "title": "论文实证分析：AI 跨国差异的人力资本约束",
  "description": "Dalu 和 DeepSeek V4 Pro 协作完成国际贸易实证论文。从固定效应出发，经内生性诊断，转向面板工具变量法。",
  "created": "2026-07-19T10:30:00Z",
  "status": "实证策略选型中",
  "next": ["弱工具变量检验", "2SLS 回归", "写实证结论"],
  "participants": [
    { "name": "Dalu", "role": "定义者" },
    { "name": "DeepSeek V4 Pro", "role": "分析策划师" }
  ],
  "branches": 1,
  "decisions": 1,
  "files": ["main_data.dta", "iv_spec.md", "summary.json", "trail.jsonl"],
  "provenance": [
    {
      "action": "created",
      "who": "Dalu (a1b2c3d4)",
      "when": "2026-07-19T10:30:00Z",
      "signature": "ECDSA-P256:base64..."
    }
  ]
}
```

### 5.3 edits.json 差异记录

```jsonc
{
  "edits": [
    {
      "sequence": 1,
      "who": "Dalu (a1b2c3d4)",
      "when": "2026-07-19T10:30:00Z",
      "action": "created",
      "changes": {
        "added": ["main_data.dta", "iv_spec.md", "summary.json", "trail.jsonl"],
        "modified": [],
        "removed": [],
        "unchanged": []
      },
      "hash_before": null,
      "hash_after": "sha256:abc123...",
      "signature": "ECDSA-P256:base64..."
    }
  ]
}
```

### 5.4 签名

- **算法**：ECDSA P-256
- **默认密钥**：首次打开 Smelt Lite 时，浏览器自动生成密钥对，存于本地（OPFS）
- **导入自有密钥**：用户可导入自己的 ECDSA P-256 密钥（JSON/JWK 格式）。适用于：
  - 换浏览器/设备时恢复身份
  - 使用第三方生成的密钥（如已有 GPG 体系里的身份）
  - 多设备共用同一签名身份
- **导出密钥**：浏览器生成的密钥可导出为 JSON 文件备份
- **打包时选择**：打包面板显示当前使用的身份。用户可选择切换密钥或导入新的
- **链式**：每条 edits 条目签名前一条的 hash。改任何一条 → 链断裂 → 导入时报 ⚠️

来自旧 Smelt 的 `features/identity/index.ts` 已包含 `exportIdentityKey` 和 `importIdentityKey`，直接复用。

### 5.5 加密（可选）

- **算法**：AES-256-GCM + PBKDF2（10 万次迭代）
- **方式**：用户设密码。加密 artifacts/ 目录下的文件
- **与签名叠加**：先签名，再加密

## 六、技术约束

### 6.1 Smelt Lite 不做什么

- 不调 AI。提示词是给用户的，AI 在别处跑
- 不管理项目。一次打包一个 .card
- 不做实时协作。签名链记录的是先后关系，不是并发关系
- 不做 OPFS 持久化。关掉页面数据消失，导出才持久化

### 6.2 Smelt Lite 做什么

- 拖入 summary.json → 解析 → 展示预览
- 拖入素材文件 → 哈希 → 加入文件清单
- 打包 → 自动组装 manifest + edits + signature → 下载 .card
- 拖入 .card → 解包 → 展示导入视图（摘要 + 来源链 + 文件 + 签名验证）
- ECDSA 密钥管理：自动生成 / 导入自有 / 导出备份
- 隐私扫描（API Key、邮箱、手机号检测）
- 中英双语
